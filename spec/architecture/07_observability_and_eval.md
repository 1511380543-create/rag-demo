# 监控与测评设计

> 本文档定义 RAG 服务的监控（Monitoring）与测评（Evaluation）能力设计。
> 适用场景：可观测性建设、检索质量量化、迭代前后效果对比。
> 本页是监控与测评的权威设计文档，接口细节见 `01`，数据表见 `02`，流程见 `03`。

## 1. 设计目标

- 监控：在线持续观察检索行为，暴露延迟、召回、分数分布等关键指标
- 测评：离线量化检索质量，基于评测集计算标准检索指标，支持迭代对比
- 二者共同支撑核心诉求：提高检索质量

## 2. 能力边界

- 监控与测评均只针对检索链路（`/rag/query` 及其底层检索），不涉及答案生成
- 存储统一落 MySQL，与现有 `rag_chunks` 存储风格保持一致
- 监控写入采用请求内同步写库，不引入异步队列与降级逻辑
- 评测为离线主动触发，不在查询链路内自动执行

## 3. 监控能力（Monitoring）

### 3.1 采集点

在 `/rag/query` 处理链路上埋点，单次查询采集以下指标：

- 请求信息：`query` 文本、`top_k`、是否启用 `filters`
- 延迟拆分（毫秒）：`embed_ms`（向量化耗时）、`retrieve_ms`（检索耗时）、`total_ms`（总耗时）
- 召回情况：`retrieved_before_filter`、`retrieved_after_filter`、`is_empty_recall`
- 分数分布：`top_score`、`min_score_value`、`avg_score`（基于召回 contexts 计算）
- 结果状态：`error_code`（成功为空，失败记录统一错误码）
- 时间戳：`created_at`

> 说明：本轮监控只负责“记录分数”，不做任何基于分数的过滤或降级；分数记录是后续“阈值过滤 + 分数降级”能力的数据基础（见 §5）。

### 3.2 存储

- 写入表：`rag_query_logs`（见 `02`）
- 写入时机：查询处理结束后（无论成功或失败）写入一条记录
- 写入失败不得影响查询主流程的返回结果（监控写库异常仅记录日志，不改变查询响应）

### 3.3 聚合与暴露

- 接口：`GET /rag/metrics`（见 `01`）
- 支持时间窗口过滤（最近 N 分钟）
- 聚合指标：总查询数、空召回率、平均/`p95` 总延迟、平均向量化延迟、平均检索延迟、平均 `top_score`

## 4. 测评能力（Evaluation）

### 4.1 评测集

- 每条评测样本包含：`case_id`（业务用例 ID）、`query_text`、ground truth、可选 `top_k`
- ground truth 同时支持两种标注方式（可单独或同时使用）：
  - `relevant_chunk_ids`：chunk 级精确标注（严格，命中以 `chunk_id` 判定）
  - `expected_keywords`：关键词命中标注（宽松，命中以关键词是否出现在召回文本判定）
- 判定优先级：若样本同时提供两类标注，`relevant_chunk_ids` 用于计算 chunk 级指标，`expected_keywords` 用于关键词命中判定，两者独立记录

### 4.2 评测指标

以单次查询召回的 `top_k` 结果为基础，计算：

- `hit@k`：命中标注为 `1`，否则为 `0`
- `recall@k`：命中的相关 chunk 数 / 标注相关 chunk 总数（仅 `relevant_chunk_ids` 模式）
- `mrr@k`：首个命中相关结果的倒数排名
- `ndcg@k`：归一化折损累计增益
- `latency_ms`：单条样本检索耗时

轮次汇总指标为逐条指标的平均值。

> 相关度粒度说明：本轮采用二值相关度（相关=1 / 不相关=0），命中通过 `relevant_chunk_ids` 或 `expected_keywords` 判定。
> `ndcg@k` 因此按二值相关度计算：能体现“命中且排名靠前更好”，但暂不区分“非常相关/稍微相关”。
> 分级相关度（如 0-3 分）为后续可选增强，届时需扩展评测集标注。

### 4.3 存储

- 评测集表：`rag_eval_dataset`
- 评测轮次汇总表：`rag_eval_runs`
- 评测逐条明细表：`rag_eval_run_items`
- 表结构详见 `02`

### 4.4 接口

- `POST /rag/eval/dataset`：批量新增/更新评测样本（按 `case_id` upsert）
- `GET /rag/eval/dataset`：列出评测样本
- `POST /rag/eval/run`：对评测集执行一轮评测，落库并返回汇总指标
- `GET /rag/eval/runs`：查看历史评测轮次，支持迭代前后对比

### 4.5 与现有回归的衔接

- 现有回归用例 `rag_retrieval_reg_001` 使用关键词命中判定，可平滑迁移为 `expected_keywords` 评测样本
- 现有已知差距 `rag_retrieval_empty_reg_001`（低相关查询应空召回）本轮暂不闭环，留待后续“阈值过滤 + 分数降级”能力落地时处理（见 §5）

## 5. 分数记录与后续规划（本轮不实现阈值/降级）

本轮范围：只“记录分数”，不做任何基于分数的过滤或降级。

- 记录内容：每次查询的 `top_score`、`min_score_value`、`avg_score` 落入 `rag_query_logs`
- 记录目的：积累分数分布数据，作为后续阈值与降级策略的依据
- 当前行为：召回结果全部按 `top_k` 返回，不因分数被丢弃，保持现有检索行为不变

后续规划（不在本轮实现）：

- 阈值过滤：引入可配置分数阈值，低于阈值的召回结果不进入 `contexts`，用于闭环 `rag_retrieval_empty_reg_001`
- 分数降级：结合分数分布做检索质量降级策略（如整体分数偏低时给出降级提示或调整返回策略）
- 上述能力落地时再补充配置项、请求参数与对应约束，本轮不预留请求字段

## 6. 可观测性方案选型与演进

### 6.1 方案分层

- 自建埋点：本项目当前方案，查询指标写入 `rag_query_logs`，通过 `/rag/metrics` 聚合暴露
- 通用可观测：OpenTelemetry + Prometheus/Grafana，擅长系统健康指标（延迟、QPS、错误率、告警），不理解“检索质量”
- LLM 原生可观测：Langfuse、LangSmith、Phoenix 等，擅长链路 trace、检索/生成分步可视化、评测集管理与迭代对比

主流分工：系统监控用通用框架，质量测评用 LLM 原生框架，二者互补而非二选一。

### 6.2 本阶段选择

- 维持自建埋点：先把“指标定义 + 评测闭环”跑通，框架只是搬运工，指标设计才是核心
- 预留抽象埋点点：将“写一条查询监控记录”收敛到单一函数/接口（如 `record_query_trace(...)`），未来接入外部框架时只在此处新增 sink，不改业务逻辑，符合 §7 不改既有语义的约束

### 6.3 演进时机（明确触发点：添加答案生成层）

- 触发条件：当项目从“纯检索”演进到“检索 + 答案生成（LLM 生成层）”时，启动可观测性升级
- 原因：引入生成层后，单靠自建的检索指标不足以定位问题，需要 query -> 检索 -> （重排）-> 生成 的完整链路 trace，以及生成质量、Token 成本、prompt 版本等维度
- 升级方向：
  - 接入 LLM 原生可观测框架（优先 Langfuse，支持开源私有化部署，规避数据出域）
  - 在预留的 `record_query_trace(...)` 处新增框架 sink，与现有 MySQL 记录并存，不替换、不回退
  - 评测层可将现有 `rag_eval_*` 概念对应到框架的数据集评测，逐步迁移或双写
- 边界：在添加答案生成层之前，不引入任何外部可观测框架，保持当前自建方案不变

## 7. 关键约束

- 监控与测评均不得修改现有 `/rag/chunks`、`/rag/index/build`、`/rag/query`、`/rag/health` 的既有成功语义
- 本轮不引入分数阈值过滤与降级逻辑，仅记录分数，保持默认检索行为不变
- 测评仅读取当前内存索引进行检索，不触发索引重建
- 本轮不引入任何外部可观测框架，演进时机以“添加答案生成层”为准（见 §6.3）
